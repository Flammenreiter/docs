== Secrets (env-Distribution)

=== Master `.env` im Workspace-Root

Master-`.env` liegt im Workspace-Root (gitignored). Pro Repo wird daraus eine eigene `.env` generiert:

[source,bash]
----
bash scripts/distribute-secrets.sh
----

Das Skript liest `.env` und schreibt:

* `api/.env` + `api/.env.test` (mit `TEST_*`-Varianten)
* `lore-admin/.env`, `session-gm/.env`, `session-player/.env` (nur `VITE_*` + Localhost-URLs)
* `shared/`, `docs/`, `ue5/` brauchen keine `.env`

=== `NODE_AUTH_TOKEN` als `AUTH_TOKEN` im Workspace `.env`

Damit `pnpm install` `@flammenreiter/api-types` aus GitHub Packages ziehen kann, brauchst du ein _Personal Access Token_ mit `read:packages`-Scope:

. https://github.com/settings/tokens -> _Generate new token_
. Scope: `read:packages` (fuer Konsumenten) oder `read:packages` + `write:packages` (fuer Publisher)
. In der workspace-`.env` als `NODE_AUTH_TOKEN=ghp_...` hinterlegen
. Optional in `~/.bashrc`/`~/.zshrc` exportieren fuer Shell-Sessions

[WARNING]
====
Stand 2026-05-06: Der aktuelle Token (`ghp_yCNmeK...`) wird von GitHub Packages als 401 Unauthorized abgewiesen. Vor Frontend-Sync (Phase 4c) muss ein neues PAT erstellt + verteilt werden.
====

== Per-Commit Version-Bump (Workflow Â§11)

=== Regel

Jeder funktionale Commit (`feat:`, `fix:`, `refactor:` mit User-Auswirkung) bumpt die `version` im jeweiligen `package.json`:

* `feat:` -> minor bump (1.0.0 -> 1.1.0)
* `fix:` / `refactor:` -> patch bump (1.0.0 -> 1.0.1)
* `feat!:` / Breaking-Change -> major bump (1.0.0 -> 2.0.0)

`chore:` / `docs:` / `test:` -> _kein_ Bump.

=== CHANGELOG.md pflegen

Pro Bump einen neuen Entry in `CHANGELOG.md`:

[source,markdown]
----
## [1.1.0] â€” 2026-05-06

### Added
- Neue Sound-Hooks `useSfx` + `useAmbient` (STORY-SOUND-001c)

### Fixed
- BUG-PLAYER-01: Login-Redirect zum Picker statt Onboarding
----

CHANGELOG-Format ist konsumiert vom `shared/changelog-parser.ts` -> `<ChangelogTrigger>`-Widget in den Frontends.

== Cross-Repo Lockfile-Sync (Phase-4-Convention)

=== Wann?

Nach jedem `@flammenreiter/api-types`-Update in `shared/` muessen die 3 Frontend-Lockfiles (`lore-admin`, `session-gm`, `session-player`) neu installiert + committet werden.

=== Warum?

Frontends ziehen die Types aus GitHub Packages. Ohne Lockfile-Update zeigt CI rot wegen Version-Drift.

=== Wie?

[source,bash]
----
# 1. shared bumpen + publishen (CI auto-publisht auf v*-Tag)
cd C:/Entwicklung/dnd/shared
pnpm changeset
git add . && git commit -m "feat(shared): ..."
git tag v1.14.0
git push origin main --tags

# 2. In jedem Frontend (parallel OK):
cd C:/Entwicklung/dnd/lore-admin
pnpm install   # zieht neue Version, updatet pnpm-lock.yaml
pnpm generate:api-types
git add package.json pnpm-lock.yaml
git commit -m "chore(deps): bump @flammenreiter/api-types to ^1.14.0"
git push

# (Analog fuer session-gm + session-player)
----

[NOTE]
====
*Voraussetzung in CI:* `NODE_AUTH_TOKEN` mit `read:packages`-Scope muss in jedem Frontend-Repo als GitHub-Action-Secret gesetzt sein.
====

=== Phase-4-Convention

* _Phase 4_: shared bumpen + publishen
* _Phase 4b_: api auf neue Version bumpen + push (Railway deployt)
* _Phase 4c_: 3 Frontends parallel bumpen + push (Cloudflare-Pages deployt)

=== Ein Commit pro Problem (Workflow-Default #2)

Pro atomarer Aenderung _einen_ eigenen Commit. Sofort pushen. Niemals buendeln.

* Conventional-Commits-Prefixes pflegen (`fix:`, `feat:`, `chore:`, `docs:`, `test:`, `refactor:`)
* Mehrere unabhaengige Aenderungen -> mehrere Commits
* Ausnahme: zwei Files, die _exakt dieselbe atomare Aenderung_ darstellen (z.B. Schema in `shared/` + Generated-Type-Update in Consumer), gehen in _einen_ Commit

== Supabase Setup + Migration-Pattern

=== Verbindungs-Setup

Supabase-Connection-String setzt sich zusammen aus:

* _Region_: `aws-1-eu-central-1` (Pooler)
* _Project-Ref_: `jwfbbymantiqkzxxivlc` (Beispiel)
* _Password_: aus `SUPABASE_PW` in workspace `.env`

[source,bash]
----
# Pooler-URL (Connection-Pooled)
DATABASE_URL=postgresql://postgres.<project-ref>:<SUPABASE_PW>@aws-1-eu-central-1.pooler.supabase.com:6543/postgres?pgbouncer=true&connection_limit=1

# Direct-URL (fuer Migrationen + Studio)
DIRECT_URL=postgresql://postgres:<SUPABASE_PW>@db.<project-ref>.supabase.co:5432/postgres
----

[WARNING]
====
*DATABASE_URL embedded password ist veraltet.* Aktuelle Konvention: Password als separater `SUPABASE_PW`-Wert in `.env`, in der URL nur noch placeholder. `bash scripts/distribute-secrets.sh` substituiert beim Verteilen.
====

=== Migration-Pattern

[NOTE]
====
*WICHTIG:* Auf dem Pooler funktioniert `prisma migrate dev` nicht (Pooler-Protokoll inkompatibel mit shadow-DB-Erstellung). Workflow:
====

. Schema-Aenderung in `api/prisma/schema.prisma`
. Migration-SQL _handgeschrieben_ in `api/prisma/migrations/<timestamp>_<name>/migration.sql`
. Lokal pruefen: `pnpm db:generate` (Prisma Client refresh)
. Migration deployen: `pnpm db:migrate:deploy` (nutzt `migrate deploy`, nicht `migrate dev`)
. Verifizieren: Supabase Studio -> SQL Editor -> Schema pruefen

[source,bash]
----
# Beispiel-Workflow
cd C:/Entwicklung/dnd/api

# 1. schema.prisma editieren
# 2. Migration anlegen
mkdir -p prisma/migrations/$(date +%Y%m%d%H%M%S)_add_active_campaign_id
cat > prisma/migrations/.../migration.sql <<'SQL'
ALTER TABLE play_groups
ADD COLUMN active_campaign_id UUID REFERENCES campaigns(id) ON DELETE SET NULL;
SQL

# 3. Generate + Deploy
pnpm db:generate
pnpm db:migrate:deploy
----

== Railway Deploy + DATABASE_URL bei Token-Rotation

=== Auto-Deploy

`api/main` push -> Railway deployt automatisch via GitHub-Integration. Build-Output in Railway-Dashboard sichtbar.

=== DATABASE_URL Update bei Token-Rotation

Wenn Supabase-Password rotiert wird (oder eines durch Sicherheits-Vorfall ungueltig wird):

. Workspace `.env` -> `SUPABASE_PW=<neuer-wert>`
. `bash scripts/distribute-secrets.sh` (lokale `.env`s aktualisieren)
. Railway Dashboard -> api Service -> _Variables_:
  ** `DATABASE_URL`: Password im URL-String ersetzen
  ** `DIRECT_URL` (falls existiert): gleicher Wert, anderer Host (`db.<project-ref>.supabase.co:5432`)
. Railway redeploy + ~5min warten auf Circuit-Breaker-Reset

== Troubleshooting

[cols="1,1,2"]
|===
| Symptom | Ursache | Fix

| `403 Forbidden` bei `pnpm install` von `@flammenreiter/api-types` | `NODE_AUTH_TOKEN` nicht gesetzt oder ungueltig | PAT mit `read:packages` setzen (siehe `LAPTOP-SETUP.md` Schritt 2b)
| `Cannot destructure property 'package' of 'node.target'` bei `npx gitnexus` | npm 11.12.0 Arborist-Bug | `npm install -g gitnexus@1.6.3` und direkt aufrufen
| Frontend zeigt CORS-Fehler | `.env`-URLs zeigen nicht auf `http://localhost:3001` | `bash scripts/distribute-secrets.sh` neu laufen lassen
| `pnpm install` warnt vor Lockfile-Mismatch | `pnpm`-Version weicht vom `packageManager`-Field ab | exakt `pnpm@10.32.1` installieren
| Prisma Client fehlt im Backend | `pnpm db:generate` vergessen | im `api/`-Ordner ausfuehren
| `prisma migrate dev` haengt / failt | Pooler-Verbindung inkompatibel | _migrate deploy_ mit handgeschriebenem SQL nutzen
| Railway-Deploy crashed mit `ECIRCUITBREAKER` | DATABASE_URL Password stale | Railway env update + Redeploy
| Smoke-Job rot, "Vite error overlay" | Crash in einer Route | Stacktrace im hochgeladenen `playwright-report/` anschauen
| Smoke-Job rot, "Got redirected to /login" | Mocked-Auth-Session verworfen (lore-admin) | `tests/e2e/helpers/mock-auth.ts` pruefen
|===
