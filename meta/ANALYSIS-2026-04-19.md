## Teil 2 — Cross-Repo-Diskrepanzen (1M-Context-Findings)

Dieser Teil konnte nur mit repo-übergreifender Simultan-Analyse erstellt werden (Opus 4.7 mit 1M-Context). Er ist das zentrale neue Ergebnis dieser Session.

### D1 — Massive Shared-Types-Version-Drift

| Repo | `@flammenreiter/api-types` Version | Delta zu 1.12.0 |
|------|-----------------------------------|----------------|
| `shared` (publiziert) | **1.12.0** | — |
| `api` | `^1.6.0` | **6 Minor-Versionen zurück** |
| `lore-admin` | `^1.6.0` | 6 zurück |
| `session-player` | `^1.7.0` | 5 zurück |
| `session-gm` | `1.9.1` (exact pin!) | 3 zurück |

**Konsequenz:** Die in 1.10.0+ hinzugefügten Schemas (CharacterResponse mit languages/proficiencies/spells, RandomTable, CampStorage, Subclass, Foundry, Contested Rolls, Durability, alle Session-Lifecycle-Schemas, Session-Planning) sind in keinem Konsumenten verfügbar. Consumer redefinieren Typen lokal → Drift eskaliert. Session-gm pinnt sogar exakt statt Caret → keine automatischen Updates.

**Lösung:** Versionen in allen 4 Konsumenten synchron auf `^1.12.0` anheben, danach `pnpm generate:api-types` laufen lassen und Kompilationsfehler systematisch abarbeiten.

### D2 — OpenAPI-Response-Typen sind `unknown`

Der größte technische Schuldpunkt des Projekts. Begründet ausführlich in `session-gm/API-GAP-ANALYSE.md`:

- `GET /characters` → `application/json: unknown`
- `GET /characters/{id}` → `application/json: unknown`
- `GET /world/npcs` → `unknown`
- `GET /world/creatures` → `unknown`
- `GET /sessions/{id}` → `unknown`
- `GET /sessions/{id}/scenes` → `unknown`
- und ~70 weitere GET-Endpoints

**Root Cause:** `@hono/zod-openapi` setup im api-repo definiert Request-Bodies mit Zod-Validatoren, gibt aber bei Success-Responses oft `z.unknown()` zurück oder keine explizite Response-Schema-Registration. Das ist im `api/TODO.md` als **L2** dokumentiert: "z.unknown() Response-Schemas in ~15 Route-Dateien".

**Konsequenz:** Alle 3 Frontends müssen Response-Typen lokal definieren → garantierte Drift-Quelle. `session-gm` hat das in `API-GAP-ANALYSE.md` akribisch dokumentiert (~30 Types zu konsolidieren in `@shared/api/types/`). Gleiches Problem in `session-player` und `lore-admin`, dort nicht explizit dokumentiert.

**Lösung:** Jeder Route im `api`-Repo braucht ein explizites Zod-Response-Schema. Die Zod-Schemas werden zentral in `shared` (dem `@flammenreiter/api-types`-Package) gehostet und von api importiert — damit ist der Kreis geschlossen.

Aus `session-gm/TODO.md` (Abschnitt 1):

### D5 — WebSocket-/Realtime-Stack-Inkonsistenz

| Repo | WebSocket-Client | Anmerkung |
|------|-----------------|-----------|
| `api` | `ws@^8.20` + `@hono/node-ws` | Eigener WS-Endpoint `/ws/*` |
| `session-player` | `socket.io-client@^4.7` | **Inkompatibel mit bare ws** |
| `session-gm` | — | Kein WS-Client! Nutzt Polling laut SPRINT-STATUS |
| `lore-admin` | — | Kein Realtime |

**Konsequenz:** session-player-WebSocket würde gegen den api-WS-Endpoint aktuell nicht funktionieren (socket.io-Handshake vs bare ws). Entweder ist socket.io-client unbenutzt (Dead-Dependency), oder es gibt einen undokumentierten zweiten Realtime-Kanal (vermutlich **Supabase Realtime** — `session-player` nutzt das nachweislich für GM-Prompts und Scenes).

**Lösung:** Klärung: Supabase Realtime für alle Live-Events? Oder eigener `/ws`-Endpoint (dann socket.io entfernen und auf `ws` vereinheitlichen)? Entscheidung als ADR in `orchestrator/docs/decisions/` festhalten.

### D6 — Stack-Divergenz zwischen session-gm und session-player

Obwohl parallel strukturiert (FSD, React 19, TanStack, Tailwind), divergieren sie stark:

| Feature | session-gm | session-player |
|---------|-----------|----------------|
| Vite-Version | 6 | **5** |
| HTTP-Client | axios | axios + **fetch** |
| State (UI) | React useState | **zustand** |
| 3D | — | Three.js + @react-three/fiber + @react-three/cannon |
| Maps | — | Leaflet + react-leaflet |
| UI-Komponenten | radix-ui (Bundle) | @base-ui/react + shadcn (v4!) |
| Realtime | keins (Polling) | socket.io-client + Supabase Realtime |
| Charts | D3 | D3 + SpiderChart custom |

**Konsequenz:** Keine Code-Wiederverwendung möglich. Gleiche Komponenten (HP-Bar, Avatar, SpiderChart, DispositionDisplay) müssten zweimal gepflegt werden. Keine gemeinsame UI-Library (ansatzweise wäre das `shared/ui/`, das aber nicht existiert — `shared/` enthält nur Contracts, keine React-Komponenten).

**Lösung:** `shared-ui` Package erwägen (eigenes Repo oder Subpackage in `shared/`), oder zumindest die Key-Komponenten (SpiderChart, HpBar, DispositionBar, EmotionIcon, TraitSpectrumBar) zwischen den beiden React-Apps angleichen. Vite-Version vereinheitlichen (6).

## Teil 3 — Architektur-Kritik & Root-Cause-Analyse

### A1 — Der eigentliche Schmerzpunkt: OpenAPI-Response-Schema-Disziplin

Fast alle anderen Probleme (D2, D4, D5 Enum-Drift, D7 UC-Drift, D10 Model-Drift) sind Symptome eines einzigen Root-Cause: **Es gibt keinen etablierten, gelebten Prozess, nach dem API-Änderungen zwingend durch `shared` propagieren.**

Das `project-standards.md` beschreibt zwar den idealen Prozess (Shared → API → Consumer), aber:

- Die OpenAPI-Spec in `shared/openapi/*.yaml` ist **minimal** (nur auth/characters/sessions), nicht die vollständige API.
- Die echten Schemas werden via `@asteasolutions/zod-to-openapi` **aus api-Zod-Schemas** generiert, nicht aus `shared/openapi/`.
- Konsumenten generieren Typen per `openapi-typescript` aus — woher? — vermutlich dem Live-API-`/openapi.json`.
- Es fehlt ein klarer Pfad: "Neue Route im api → Zod-Schema in shared → automatisch in allen Konsumenten verfügbar."

**Strategische Entscheidung nötig:** Ist `shared` die **OpenAPI-First-SSoT** (YAML → TS-Gen, OpenAPI treibt Code) oder die **Zod-First-SSoT** (Zod in shared → zu OpenAPI generiert → api importiert Zod direkt)?

Die aktuelle Hybrid-Setup mischt beides und funktioniert dadurch nirgendwo sauber. Der Zod-First-Ansatz wäre einfacher, weil alle Schema-Logik TypeScript-seitig lebt und die Zod-Regex-Validatoren erhalten bleiben. OpenAPI fällt dann als Nebenprodukt an.

### A2 — session-gm ist gegenüber session-player deutlich "rougher"

Das Sprint-Ergebnis vom 2026-04-04 ("17.000 Zeilen Mock-Code entfernt, Neuaufbau in 1 Sprint") erklärt die Hastigkeit: 14 Routen in einer Woche implementiert. Die `REVIEW-BERICHT.md` dokumentiert 68 Findings (21 gefixt). Das UI-Qualitätsniveau liegt deutlich unter session-player (siehe duplizierte Komponenten, Raw HTML, hardcoded 1497 DR).

Die GM-App ist aber die **wichtigste** für die Freitags-Session. Hier ist ein Quality-Sprint nötig, bevor neue Features gebaut werden — sonst baut man auf instabilem Fundament.

### A3 — UE5 ist noch Skelett

Das UE5-Repo ist gut dokumentiert (README ausführlich), aber die tatsächliche Content-Basis fehlt. Da der User UE5 explizit aus der Umsetzungsphase ausschließt, ist das ok. Wichtig: UE5 wird in `docs/ROADMAP.adoc` als Phase 3 (nach Web-MVP) eingestuft — das passt zur aktuellen Situation.

### A4 — orchestrator als zentraler Hub ist unterentwickelt

Der orchestrator-Repo ist theoretisch der Sprint-Koordinator über alle 7 anderen Repos. Tatsächlich:
- `.scrum/{backlog,in-progress,review,done}/` — Struktur da, aber unbekannt wie voll
- ADR-Verzeichnis leer
- OpenAPI-Contracts-Verzeichnis leer
- `ultraplan-flammenreiter/` ist die faktische Realität — inoffizielle Einzel-Session-Planung, nicht orchestriert
- CLAUDE.md und AGENTS.md referenzieren PO/SM/Designer-Modelle, aber keine Anbindung zur Realität (welche Stories laufen gerade?)

Der orchestrator ist heute eine **statische Dokumentation** über idealtypische Prozesse, nicht ein operativer Hub.

## Teil 7 — Offene Fragen vor Umsetzungsstart

Bevor mit Block 1 begonnen wird, brauche ich Klärung zu:

1. **Commits pro UC/Feature:** Die Regel `project-standards.md` sagt "1 Commit nach Merge" — aber sollen wir **pro UC/Fix einen eigenen Commit** machen, oder reicht **pro Block ein Sammel-Commit mit mehreren Files**? (User hat "Pro abgeschlossenem UC/Feature ein Commit" im Ablauf-Question ausgewählt → ich interpretiere: pro UC einen.)

2. **Cloudflare Worker (P3-9):** Wirklich löschen? Es gibt ggf. noch Traffic drauf, wenn DNS nicht umgestellt ist. Brauche Bestätigung vor Abschaltung.

3. **Enum-Normalisierung (P1-3):** Wir bringen Breaking-Change-Gefahr mit (z.B. `ACTIVE` → `active` im EncounterStatus würde API-Response-Format brechen). Soll ich:
   - Variante A: Shared definiert beide Versionen, API gibt uppercase raus (wie heute), Konsumenten mappen beim Lesen → niedrigeres Risiko, aber Normalisierung nur nominell
   - Variante B: Migration (api + alle Konsumenten gleichzeitig auf lowercase), DB-Enum bleibt uppercase → API transformiert → klarer aber breaking
   - Variante C: DB-Migration auch → am saubersten, aber echtes Release-Risiko
   *Empfehlung: **Variante B** mit Deployment innerhalb eines Tages, damit alle Komponenten gleichzeitig updaten.*

4. **WebSocket (P1-7):** Die socket.io-Dependency in session-player deutet an, dass mal ein eigener /ws-Server geplant war. Aktuell nutzt session-player aber Supabase Realtime (laut Code-Evidenz in UC-IMPLEMENTIERT für GM-Prompts). Soll ich einfach socket.io entfernen (Dead-Dep) und auf Supabase Realtime setzen? Oder gibt es Pläne, von Supabase Realtime wegzugehen?

5. **React 19 RC in lore-admin:** Der RC-Build ist spezifisch (`19.0.0-rc-66ef8a70-20250409`). Gibt es einen Grund (offener Bug in stable), oder können wir direkt auf stable?

6. **UC-Nummerierungs-Migration (P1-5):** Soll die lore-admin-UC-MEC-74..80 Umnummerierung auf die kanonischen Nummern als **Breaking-Change-Refactor** (alle References aktualisieren) oder als **Alias-Mapping** (lore-admin behält interne Nummern + Mapping-Tabelle zu kanonischen) erfolgen? Refactor ist sauberer, aber invasiv.

7. **Tests (P1-6):** Für die Smoke-Tests brauche ich Zugriff auf eine lokale Postgres-Instanz. Ist `docker-compose up -d` aus api-repo funktionsfähig, oder nutzt ihr Supabase-Cloud für alle Tests? Frage ist wichtig, weil Mock-basierte Tests (Prisma-Mock) anders laufen als echte DB-Integration-Tests.

## Teil 8 — Zusammenfassung in einem Satz

Das Flammenreiter-Projekt hat **ein gesundes technisches Rückgrat** (7 kohärent strukturierte Repos, klare Rollenteilung, >200 Backend-Endpoints) mit **drei systemischen Drift-Quellen** (Shared-Types-Versionen, OpenAPI-`unknown`-Responses, Enum-Divergenzen über DB/Shared/Frontend-Layer) plus **mehreren dringenden Sicherheits-Gaps** (12 Endpoint-Gruppen ohne Ownership-Check, 0 Backend-Tests), deren systematische Behebung in ~23 Arbeitstagen ein signifikant belastbareres Fundament schafft — und danach kann man sich wieder auf reines Feature-Wachstum konzentrieren.
