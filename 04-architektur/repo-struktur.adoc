== shared (npm-Package)


Single-Source-of-Truth für Schemas, von allen Frontends + api konsumiert.


=== Struktur

[source]
----
shared/
├── src/
│   ├── schemas/              Zod-Schemas pro Domain
│   │   ├── character.ts
│   │   ├── session.ts
│   │   ├── scene.ts
│   │   ├── npc.ts
│   │   └── ...
│   ├── enums.ts              ~50 feste Enums
│   ├── openapi.ts            Zod-to-OpenAPI Config
│   └── index.ts              Exports
│
├── package.json
│   ├── name: @flammenreiter/api-types
│   └── publishConfig: registry npm.pkg.github.com (GitHub Packages)
├── tsconfig.json
├── tsup.config.ts            Build-Config (CJS + ESM + Types)
└── .changeset/                Changesets für SemVer-Bumps
----


=== Publishing-Workflow

. Schema-Änderung in `shared/src/schemas/…`
. `pnpm changeset` → Bump-Typ + Beschreibung
. PR → Review → Merge auf `shared/main`
. Tag `vX.Y.Z` → CI publisht auf GitHub Packages
. Consumer-Repos bumpen `@flammenreiter/api-types` in `package.json`

Aktuelle Version: `v3.19.x` (2026-05-11).

== Frontends (3× FSD-React-SPA)


Alle drei Frontends (`lore-admin`, `session-gm`, `session-player`) folgen **Feature-Sliced Design** (FSD), nicht der klassischen `pages/components/hooks/store/`-Struktur.


=== FSD-Layer-Hierarchie

[source]
----
src/
├── app/                  Globaler Setup (Router-Config, Providers, Theme)
├── pages/                Route-Endpoints (eine Page pro Route)
├── widgets/              Komplexe UI-Blöcke (CharacterHeader, SessionPanel)
├── features/             User-Aktionen (RollDice, SwitchScene, EquipItem)
├── entities/             Business-Objekte (Character, Session, Scene, NPC)
├── shared/               Reusable Layer (ui/, api/, lib/, config/)
│   ├── ui/               Atoms + Molecules (Button, Card, Modal, …)
│   ├── api/              HTTP-Client + Generated-API-Types
│   ├── lib/              Pure Utilities
│   └── config/           Constants, Env, Routes
│
└── main.tsx              Entry
----

WARNING: Imports nur Top-Down: `app → pages → widgets → features → entities → shared`. Niemals andersrum. Lint-Rule via `eslint-plugin-boundaries`.


=== session-gm — Eigenheiten

* _D3.js_ für Session-Planer-Graph + Spider-Charts
* _Radix-UI_ + Tailwind als Primary-Component-Stack
* _Supabase Realtime_ für Live-Session-Sync zwischen GM und Spielern
* _Tiptap_ für Rich-Text-Editing (Session-Notes, NPC-Bios)


=== session-player — Eigenheiten

* _Three.js + R3F_ für 3D-Würfel und potenziell 3D-Charakter-Viewer
* _Leaflet_ für Karten + Travel-Routes
* _Supabase Realtime_ für Live-Updates (Szenen-Wechsel, NPC-Dispositions)
* _Zustand_ für Frontend-State + TanStack-Query für Server-State
* _shadcn4 + base-ui_ statt Radix (Variation)


=== lore-admin — Eigenheiten

* Welt-Daten-Editor: NPCs, Items, Spells, Lore
* Leaflet-Editor für Karten + POIs
* Tiptap für Rich-Text-NPC-Bios
* Radix + Tailwind

== Lockfile-Sync-Workflow (3 Frontends + api)


Wenn `@flammenreiter/api-types` (aus `shared/`) ein neues Release hat, **müssen** die 4 Consumer-Lockfiles neu installiert werden:

[source,bash]
----
# In jedem Consumer-Repo:
cd api
pnpm install                    # zieht neue api-types-Version
pnpm test
git add pnpm-lock.yaml package.json
git commit -m "chore: bump @flammenreiter/api-types to vX.Y.Z"

# Analog für lore-admin, session-gm, session-player
----

CI braucht `NODE_AUTH_TOKEN` mit `read:packages`-Scope als GitHub-Action-Secret (siehe link:../10-infrastruktur/[10-infrastruktur/]).

'''

== Cross-Repo-Deploy-Order


Feature über shared + api + Frontends:

. _shared_ — Schema-Change, Version-Bump, Tag, CI publisht
. _api_ — Import neue Schemas, Routes anpassen, Push → Railway deployt
. _Frontends (parallel)_ — `@flammenreiter/api-types` bumpen, `pnpm generate:api-types`, UI anpassen
. _ue5_ — Last (falls betroffen)

Siehe `workspace-docs/guides/setup.md` für Detail-Workflow.
