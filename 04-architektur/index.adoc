
= Architektur und Technologie


_Stand: {revdate}._ Architektur-Übersicht des Flammenreiter-Systems — Live-Stack der 8 Repositories (7 aktiv + ue5 dormant).

NOTE: Diese Doku spiegelt den **aktuellen** Live-Stack (Sprint 2026-05-x). Frühere Versionen referenzierten Supabase PostgREST, Vercel und Expo — siehe `meta/AUDIT-2026-03-25.adoc` für die historische Vision.

'''


== Überblick


Flammenreiter nutzt eine moderne, polyglotte Architektur mit mehreren spezialisierten Repositories statt eines Monorepos. Die wichtigsten Live-Komponenten:

* _Backend_: Hono 4.6 + Prisma 6 + PostgreSQL (über Supabase) — kein PostgREST mehr als Primär-Layer
* _Frontends_: 3 separate React-19-SPAs (`session-gm`, `session-player`, `lore-admin`) mit Feature-Sliced Design (FSD)
* _Shared Contracts_: `@flammenreiter/api-types` via GitHub Packages (Zod 4 + OpenAPI Spec)
* _Mobile_: UE5 5.4+ C++/Blueprints (dormant, nicht aktiv)
* _Auth_: Supabase Auth (JWKS-verified im api über JWT)
* _AI / Bild_: fal.ai FLUX-Pipeline für Portraits/Maps

'''


== Backend-Architektur


Hono 4.6 als alleiniger API-Layer (kein PostgREST-Mix mehr). Prisma 6 als ORM gegen Supabase Postgres. Detaillierte Beschreibung siehe link:./backend.adoc[backend.adoc].

Wichtige Charakteristika:

* TypeScript-native, Zod-native Validierung
* Edge-kompatibel (initial auf Cloudflare Workers gelauncht, jetzt auf Railway)
* OpenAPI-3.1-Spec automatisch generiert über `@hono/zod-openapi`
* JWT-Auth über Supabase JWKS (`/health` + `/version` ohne Auth)

'''

== Shared Types & Schemas


`@flammenreiter/api-types` (publisht aus `shared/`) ist die Single-Source-of-Truth für API-Schemas:

* _Zod 4_ Schemas für Validierung in api + 3 Frontends
* _OpenAPI 3.1_ Spec automatisch via `@asteasolutions/zod-to-openapi`
* _TypeScript Types_ via `openapi-typescript`
* _Versioning_: SemVer + Changesets → GitHub Packages auf `v*` Tag

Detaillierte Beschreibung siehe link:./shared-types.adoc[shared-types.adoc] und link:./ai-prompt-system.adoc[ai-prompt-system.adoc] (ADR-017 Unified-Prompts).

Aktuelle Version (2026-05-11): `v3.19.x` live.

'''


== Auth & Berechtigung


* _Auth-Provider_: Supabase Auth (Email/OAuth) — JWT-basiert
* _api-Verify_: JWKS-Endpoint von Supabase wird in Hono-Middleware konsumiert
* _RLS_: aktuell nicht primär — Berechtigung läuft über api-Layer-Checks (campaign_id Scoping)
* _Tokens_: JWT in Frontend-Storage, kein Refresh-Token-Cookie

== Design-Prinzipien


=== 1. Separation of Concerns

* _DB_: Datenspeicherung, FK-Constraints, Migrations (Prisma)
* _Business Logic_: Hono-Routes + Services, **nicht** in Stored Procedures
* _Auth_: Supabase Auth JWT-Layer, in api per Middleware verifiziert
* _Frontends_: Pure UI + State (Zustand) + Server-State (TanStack Query)


=== 2. Type Safety durch die ganze Chain

[source,typescript]
----
Prisma Schema → Generated DB-Types
              → Zod Schemas (in shared/)
              → OpenAPI Spec (auto-gen via @hono/zod-openapi)
              → TypeScript Types (openapi-typescript)
              → Konsumiert in api + 3 Frontends
----

Jeder Layer validiert; niemand vertraut auf vorherige Ebene.


=== 3. Multi-Repo statt Monorepo

* _Grund_: Klare Boundaries (api / shared / 3 Frontends / docs / ue5), unabhängige Deploys
* _Lösung_: 8 Repos + zentrales `workspace` für Orchestration
* _Cross-Repo-Workflow_: shared-Bump → api consumes → Frontends consumes (lockfile-sync)


=== 4. No Stored Procedures (Mostly)

Geschäftslogik gehört in Hono, nicht in SQL. Ausnahmen:

* Trigger für Auditing (`session_mechanical_log`)
* Supabase Realtime Event Publishing


=== 5. Progressive Reveal

Szenen-Elemente werden progressiv revealed (Title → Description → Image → NPCs). DB speichert JSON mit `is_revealed`-Boolean. Realtime-Sync zwischen GM und Spielern via Supabase Channels.

'''


== Typ-Flow


[source]
----
Prisma Schema (api/prisma/schema.prisma)
  ↓

  ↓

----
