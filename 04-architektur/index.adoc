
= Architektur und Technologie


Architektur-Übersicht des Flammenreiter-Systems mit Backend-Schichten, Repo-Struktur und Shared Types.


== Überblick


Das Flammenreiter-Projekt nutzt eine moderne, polyglotte Architektur mit mehreren spezialisieren Repositorien statt eines Monorepos. Das Backend folgt einer 3-Schichten-Hybrid-Architektur:

. _Supabase PostgREST_ — Auto-CRUD für ~70% der Requests
. _Hono Application Layer_ — Business Logic, Validierung, Workflows in TypeScript
. _Supabase Edge Functions_ — Externe Integrationen (KI, Discord, etc.)

== Typ-Flow


----
PostgreSQL Schema
  ↓
supabase gen types (Supabase CLI)
  ↓
@flammenreiter/shared-types (TypeScript)
  ↓
Zod Schemas (Validierung)
  ↓
npm publish @flammenreiter/*
  ↓
Alle Apps importieren
----


== Authentifizierung & RLS


* _Auth_: Supabase Auth (Email/OAuth)
* _RLS_: Row Level Security auf `auth.uid()` und `campaign_id`
* _Tokens_: JWT in Session + Refresh Tokens in HttpOnly Cookies

== Design-Prinzipien



=== 1. Separation of Concerns


* _DB macht nur_: Datenspeicherung, FK-Constraints, RLS, Realtime Events
* _Business Logic_: NICHT in Triggers/Stored Procedures, sondern in Hono Services
* _Edge Functions_: NUR für externe Integrationen (KI, Webhooks)


=== 2. Type Safety durch die ganze Chain


[source,typescript]
----
PostgreSQL → TypeScript Types → Zod Schemas → Runtime Validierung
----

Jeder Layer validiert; niemand vertraut auf vorherige Ebene.


=== 3. Multi-Repo statt Monorepo


* _Grund_: 200 DB-Tabellen + 5 Apps → zu viel für ein Repo
* _Lösung_: Separate Repos mit shared npm packages
* _Vorteil_: Unabhängige Deployments, klare Boundaries


=== 4. No Stored Procedures (Mostly)


Geschäftslogik gehört in Hono, nicht in SQL. Ausnahmen:
* RLS Policies (Sicherheit)
* Trigger für Auditing (session_mechanical_log)
* Realtime Event Publishing


=== 5. Progressive Offenbarung


Szenen-Elemente werden progressiv revealed (Title → Description → Image → NPCs). DB speichert JSON mit `is_revealed` Boolean.
