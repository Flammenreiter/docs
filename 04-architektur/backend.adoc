== Hono Application Layer


=== Zweck

Single-Layer TypeScript-Backend für **alle** Business-Logic und CRUD-Operations. Keine Auto-CRUD-Schicht — alles geht über typisierte Routes.


=== Warum Hono?

[cols="1,1,1,1",options="header"]
|===
| Kriterium | Hono | NestJS | Express

| _Bundle Size_ | ~30 KB | 500 KB+ | ~50 KB
| _Cold Start_ | <100ms | 500ms+ | ~100ms
| _TypeScript_ | Native | Native | Addon
| _Zod Integration_ | Built-in (`@hono/zod-openapi`) | Addon | Addon
| _Edge Compatible_ | Ja (Cloudflare/Deno/Bun) | Nein | Nur Node
| _OpenAPI Auto-Gen_ | Ja (über Zod) | Addon | Addon

_Entscheidung_: Hono — Cloudflare-Workers-launchable, jetzt auf Railway als Standard-Deploy (Edge-Migration jederzeit zurück möglich).

[[effect-processor]]
== EffectProcessor (Herz des Systems)


=== Zweck

Zentrale Stelle, wo alle mechanischen Modifikatoren eines Charakters gesammelt und berechnet werden. Output ist eine **transparente** Modifier-Chain mit Quellen-Breakdown.


=== Datenquellen (in Prisma-Schema)

Der EffectProcessor liest `mechanical_effects[]` aus:

. Traits (`character_trait_levels` → `trait_spectrum_effects`)
. Items (equipped in `character_inventory` → `item_special_abilities`)
. Conditions (`character_active_effects`)
. Wetter (`campaign_calendar_state` → seasons/weather → `mechanical_effects`)
. Saison (`mechanical_effects` auf seasons)
. Gelände (`campaign_travel_days` → terrain_types → `mechanical_effects`)
. Buffs/Debuffs (`character_active_effects`)
. Konsequenzen (`character_criminal_record` → `legal_consequences` → `mechanical_effects`)
. Drachen-Bond (`character_dragon_bonds` → `dragon_trait_levels` resonance)
. Race-Innate (Cantrips, Resistances, Speeds — neu in Sprint 2026-05)


=== Algorithmus

[source,typescript]
----
class EffectProcessor {
  async getModifierStack(characterId: string, target: EffectTarget): Promise<ModifierStack> {
    const effects: MechanicalEffect[] = []

    effects.push(...await this.getTraitEffects(characterId))
    effects.push(...await this.getItemEffects(characterId))
    effects.push(...await this.getConditionEffects(characterId))
    effects.push(...await this.getEnvironmentEffects(characterId))
    effects.push(...await this.getRaceInnateEffects(characterId))
    effects.push(...await this.getDragonBondEffects(characterId))

    const relevantEffects = effects.filter((e) => e.target === target)
    const stack = this.calculateStack(relevantEffects)

    return {
      finalModifier: stack.total,
      components: [
        { label: 'Base Ability', value: 3 },
        { label: 'Proficiency', value: 2 },
        { label: 'Trait (Loyal)', value: 1 },
        { label: 'Rain Penalty', value: -2 },
        { label: 'Dragon Resonance', value: 1 },
      ],
      description: 'Perception +5: Base +3, Proficiency +2, Loyal +1, Rain -2, Dragon +1',
    }
  }

  private calculateStack(effects: MechanicalEffect[]): CalculatedStack {
    // Advantage/Disadvantage cancel out (D&D Standard)
    const advantages = effects.filter((e) => e.operation === 'advantage').length
    const disadvantages = effects.filter((e) => e.operation === 'disadvantage').length
    const netAdvantage = advantages - disadvantages

    // Highest 'set' wins
    const setValues = effects
      .filter((e) => e.operation === 'set')
      .map((e) => e.value)
      .sort((a, b) => b - a)
    const base = setValues.length > 0 ? setValues[0] : 0

    // Adds stack
    const adds = effects
      .filter((e) => e.operation === 'add')
      .reduce((sum, e) => sum + e.value, 0)

    // Multiplies combine
    const multiplies = effects
      .filter((e) => e.operation === 'multiply')
      .reduce((prod, e) => prod * e.value, 1.0)

    return { base, adds, multiplies, netAdvantage, total: (base + adds) * multiplies }
  }
}
----

== Deployment (Live: Railway)


Die api ist auf Railway gehostet:

* _Trigger_: Push auf `api/main`
* _Build_: `pnpm install && pnpm build` (tsup)
* _Run_: `node dist/index.js` als persistent Node-Process
* _Migrationen_: NICHT automatisch — manuell via `pnpm prisma migrate deploy` nach Push
* _Health-Check_: `/health` (Auth-frei)
* _Custom-Domain_: `api.flammenreiter.gmolike.me`

WARNING: Railway deployt **sofort** bei Push-to-main. Vor Push CI + Tests grün haben.


=== Edge-Migration (Future-Option)

Code ist Edge-kompatibel geschrieben (Lazy-Config, kein `process.exit()`, Native-Response-Wrap — siehe link:../10-infrastruktur/[10-infrastruktur/]). Eine Re-Migration zurück auf Cloudflare Workers ist jederzeit möglich, falls Latency oder Kosten das erfordern.

== Error Handling & Logging


=== Strukturierte Errors

[source,typescript]
----
class GameError extends Error {
  constructor(
    public code: string,
    public message: string,
    public statusCode: number,
    public context?: Record<string, unknown>,
  ) {
    super(message)
  }
}

app.onError((err, c) => {
  if (err instanceof GameError) {
    return c.json({ error: err.code, message: err.message, context: err.context }, err.statusCode as ContentfulStatusCode)
  }
  console.error(err)
  return c.json({ error: 'INTERNAL_ERROR', message: 'Something went wrong' }, 500)
})
----

== Caching-Strategie


Aktuell minimal — Prisma + Supabase haben eigenes Connection-Pooling. Application-Cache:

* _In-Memory_: Pro Worker-Instance Map<string, T> mit TTL (5min Standard) — nur für teure Reads (Spell-Lookups, Item-Catalog)
* _Cache-Control Header_: `public, max-age=3600` für statische Welt-Daten (Spells, Items)
* _Realtime-Invalidation_: Supabase-Channel "character_updated" → Frontend invalidiert TanStack-Query-Cache
