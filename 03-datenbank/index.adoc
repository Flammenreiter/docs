
= Datenbank-Schema


Umfassende Referenz für die PostgreSQL-Datenbankarchitektur des Flammenreiter-Systems.


== Überblick


Das Flammenreiter-Datenbank-Schema ist eine hochgradig normalisierte (5NF) PostgreSQL-Datenbank mit PostGIS-Erweiterung für geografische Daten. Die Architektur teilt sich in drei Ebenen nach Scope und Zugriff:

* _User-Ebene_: 3 Tabellen (Profile, Kampagnen-Mitgliedschaft, Einladungen)
* _Welt-Ebene_: ~80 Tabellen (kampagnenunabhängig, alle Spieler lesen mit `player_visible`-Filter)
* _Kampagnen-Ebene_: ~10 Tabellen (RLS-isoliert pro Kampagne)
* _Charakter-Ebene_: ~15 Junction-Tabellen (Verknüpfungen zwischen Charakteren und World-Daten)

_Gesamt_: ~200 Tabellen · PostgreSQL 16+ · Supabase gehostet

== Designprinzipien



=== Feste Enums vs. Lookup-Tabellen


_FESTE ENUMS_ (Direkter ENUM-Type in PostgreSQL):
* TypeScript enum + PostgreSQL enum type + Zod z.enum()
* Für Dinge die sich praktisch nie ändern (Fähigkeitsscores, Schadenstypen, etc.)
* ~50 Enums in `@flammenreiter/shared-types/enums.ts`

_LOOKUP-TABELLEN_ (CRUD-erweiterbar):
* Eigene Tabelle mit id + name + description
* JOIN nötig
* Für Systeme die wachsen sollen (Spells, Creatures, Professionen, etc.)
* ~15 Lookup-Tabellen


=== MechanicalEffect: Der universelle Typ


Jedes Entity mit mechanischen Auswirkungen nutzt denselben Zod-Schema:

[source,typescript]
----
type MechanicalEffect = {
  target: EffectTarget        // WAS wird beeinflusst
  operation: EffectOperation  // WIE (add, set, multiply, etc.)
  value: number               // WIEVIEL
  condition?: EffectCondition // WANN (z.B. bei Regen)
  duration?: EffectDuration   // WIE LANGE
  description: string         // UI-Text für Transparenz
}
----

Verwendet in:
* Traits & Spektren
* Items & Ausrüstung
* Spells
* Race/Class-Features
* Feats
* Wetter & Jahreszeiten
* Gelände
* Folgen & Bestrafungen
* Professionen
* Dragon Bonds


=== RLS (Row Level Security)


* _User-Daten_: `user_profiles` nur eigenes Profil schreibbar, lesbar für Kampagnen-Mitglieder
* _Welt-Daten_: `player_visible` Filter auf Lesezugriff
* _Kampagnen-Daten_: `campaign_id` Isolation mit `auth.uid()` Validierung über `campaign_members`
* _Einladungen_: Erstellen nur GM, Lesen via `invite_code` für Beitritt
* _Charakter-Daten_: Lesezugriff nur für Charakter-Besitzer + GM


== Abhängigkeiten & Foreign Keys


Die Datenbankstruktur folgt einer klaren Hierarchie:

----
auth.users (Supabase managed)
  └── user_profiles (1:1)

campaign_members (User ↔ Campaign)
  ├── role: gm | player
  └── campaign_invitations (Einladungs-Lifecycle)

campaigns (Root)
  ├── campaign_members
  ├── campaign_characters
  ├── sessions
  ├── campaign_calendar_state
  └── [alle Kampagnen-Tabellen]

characters (unter campaign_characters)
  ├── character_ability_scores
  ├── character_trait_levels
  ├── character_spell_slots
  ├── character_inventory (Self-Ref für Container)
  └── [alle Charakter-Junction-Tabellen]

world_data (kampagnenunabhängig)
  ├── races, classes, spells, items, ...
  ├── [mit player_visible Flags]
  └── [RLS-protected Zugriff]
----

== Wichtige Systeme



=== Das Trait-Spektren-System


Traits sind nicht einfache Boolean-Flags, sondern kontinuierliche Spektren von -5 bis +5 mit stufengesteuerten Effekten:

----
trait_spectrums (Welt)
  ├── trait_spectrum_effects (pro Level -5...+5, mit MechanicalEffects)
  └── trait_exclusion_rules (Hard/Soft-Ausschlüsse)

character_trait_levels (Kampagnen)
  └── Dezimalwerte -5.0 bis +5.0
----

Siehe link:./schema-details.adoc[schema-details.adoc] für vollständige Dokumentation.


=== Das Items-System (10 Tabellen)


Hochgradig normalisiert mit Varianten-Support:

* _items_: Basis-Definition
* _item_rarity_variants_: 7 Seltenheitsstufen pro Item (Damage-, AC-Skalierung)
* _item_weapon_stats_, _item_armor_stats_: Typ-spezifische Stats
* _item_container_stats_: Volumen & Stealth-Modifikatoren
* _item_special_abilities_: Magische Fähigkeiten als MechanicalEffects

Siehe link:./schema-details.adoc[schema-details.adoc] für die vollständige Item-Dokumentation.


=== Das Dragon-Bond-System (9 Tabellen)


Komplette Drachen-Verwaltung mit Phasen, Care-Tasks und Equipment:

* _dragon_instances_: Einzelne Drachen mit Persönlichkeit
* _character_dragon_bonds_: Rider ↔ Dragon Beziehung
* _dragon_bond_events_: Phasen-Fortschritt und Vertrauen-Milestones
* _dragon_care_tasks_: Daily/Weekly/Seasonal Pflichten
* _dragon_equipment_: Sattel, Rüstung, Fokus-Kristalle

Siehe link:./schema-details.adoc[schema-details.adoc] für die vollständige Dragon-Bond-Dokumentation.


== Performance & Indizes


Kritische Indizes sind in der Schema-Definition enthalten:

* `(campaign_id, character_id)` für Charakter-Lookup
* `(entity_type, entity_id)` für polymorphe Abfragen
* `(created_at)` für Session-Logs
* PostGIS Spatial Indizes auf geography-Spalten


== Migrationen & Versionierung


Alle DB-Änderungen sind in `flammenreiter-supabase/migrations/` dokumentiert. Migrationen folgen dem Format:

----
migrations/
  ├── 001_initial_schema.sql
  ├── 002_add_traits_system.sql
  ├── 003_add_dragon_bonds.sql
  └── ...
----
