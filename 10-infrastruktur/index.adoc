== API Deployment (Railway)


Der `api/`-Service (Hono 4.6 + Prisma 6) wird automatisch deployed:

. Push auf `api/main`
. Railway erkennt Commit → Build (`pnpm install`, `pnpm build`)
. Prisma-Migrationen werden bei Deploy nicht automatisch angewendet (manueller Schritt via `pnpm prisma migrate deploy`)
. Health-Check auf `/health` (status + version, ohne Auth)
. Custom-Domain `api.flammenreiter.gmolike.me` aktiv via CNAME

WARNING: Push auf `api/main` triggert sofortigen Deploy. Vor Push immer Tests laufen lassen.

== Frontends (Cloudflare Pages)


Alle drei React-Frontends (`session-gm`, `session-player`, `lore-admin`) werden auf Cloudflare Pages gehostet:

* _Build_: `pnpm build` → Vite-Output nach `dist/`
* _Deploy_: GitHub Action triggert auf Push-to-main → Cloudflare Pages Build
* _Custom-Domains_: pro Frontend ein CNAME (siehe Tabelle oben)
* _Auth_: Supabase Auth (JWKS-verified im api über JWT)

CI-Workflows der drei Frontends brauchen `NODE_AUTH_TOKEN` mit `read:packages`-Scope als GitHub-Action-Secret (für `@flammenreiter/api-types` aus GitHub Packages).

'''


== Shared Types Publishing (GitHub Packages)


Schema-Änderungen fließen durch `shared/`:

. Change Zod-Schema in `shared/src/schemas/…`
. `pnpm changeset` → dokumentiert Bump (patch/minor/major) + Beschreibung
. Merge auf `shared/main` + Tag `vX.Y.Z` → CI publishes auf GitHub Packages (`npm.pkg.github.com`)
. Consumers (`api`, 3 Frontends) bumpen `@flammenreiter/api-types` in `package.json`
. In api: `createRoute({ responses: { … } })` an neue Schemas anpassen
. In Frontends: `pnpm generate:api-types` regeneriert Consumer-Types

Aktuelle Version (2026-05-11): `v3.19.x` live.

'''

== CI/CD-Hinweise


=== GitHub Packages Auth

CI-Workflows die `@flammenreiter/*` Pakete nutzen brauchen:

[source,yaml]
----
permissions:
  contents: read
  packages: read
----

Plus: Im Quell-Repo des Pakets unter **Package Settings -> Manage Actions access** das konsumierende Repo mit "Read"-Rolle hinzufügen.


=== ESLint

`typescript-eslint` muss als explizite devDependency installiert sein wenn `eslint.config.mjs` es importiert.


=== Pre-Commit-Hook Version-Bump

Jedes Code-Repo (`api`, `shared`, `lore-admin`, `session-gm`, `session-player`) hat einen `pre-commit`-Hook (`scripts/version-bump.mjs`), der vor `lint-staged` läuft und das `version`-Feld in `package.json` SemVer-konform bumpt (siehe Workflow-Default §11).

'''


== Workers-Learnings (legacy, vor Railway-Migration)


Diese Erkenntnisse stammen aus der Cloudflare-Workers-Phase des api-Repos (vor Umzug auf Railway). Sie bleiben relevant, falls einzelne Edge-Workflows künftig wieder auf Workers landen.


=== `process.exit()` vermeiden

Crasht den Worker. Stattdessen `throw new Error()` verwenden.


=== Config lazy laden

Environment Variables sind auf Workers erst nach dem ersten `fetch()` in `process.env` verfügbar. Config muss lazy geladen werden (z.B. via Proxy):

[source,typescript]
----
// FALSCH: Wird beim Import ausgefuehrt, env noch leer
const config = parseEnv(process.env)

// RICHTIG: Lazy via Proxy
export const config = new Proxy({} as Config, {
  get(_, key) {
    if (!_parsed) _parsed = parseEnv(process.env)
    return _parsed[key]
  }
})
----


=== Dynamischer `import()` — Response-Mismatch

Hono's Response ist nicht die native Workers-Response. Fix: NativeResponse vor dem Import capturen und Response wrappen:

[source,typescript]
----
const NativeResponse = Response

// ... nach app.fetch():
return new NativeResponse(res.body, {
  status: res.status,
  statusText: res.statusText,
  headers: res.headers,
})
----

'''

[cols="1,1,1,1",options="header"]
|===
| Szenario | Maßnahme | RTO | RPO

| _Frontend Bad Deploy_ | Cloudflare Pages Rollback via Dashboard | 5 min | 0 min

|===

== Security Best Practices


[cols="1,1",options="header"]
|===
| Aspekt | Maßnahme

| _Secrets Management_ | GitHub Secrets + Railway Variables; keine Secrets im Code
| _RLS Enforcement_ | (geplant — aktuell läuft Auth via api-Layer-Checks)
| _HTTPS Only_ | Alle Cloudflare/Railway-Endpoints HTTPS
| _CORS Policy_ | Whitelist explizit für gm/player/lore-domains
| _Input Validation_ | Zod-Schema-Validation überall (api + Frontends)
| _SQL Injection_ | Prisma parameterisiert Queries automatisch
|===
